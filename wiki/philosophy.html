Profiles can be created by humans signing-up to the system or created by admin. Tasks are created by admin.

Tasks are normally to track an external process that is repeated many times as independent processes. The task is just a tracking utility it does not perform the task that it tracks. (It is a 'to do' list with progress and built-in detailed instructions)

When a task is created this brings into existence the category of 'author'. An author is a profile associated with the human who created the task.

Tasks automatically start with 3 steps. These are subcategory of tasks which as a child inherits the attributes of task plus a step_order which is an integer to enable the steps to be placed in order.

Therefore on the person side we have 'profiles' which are long term. These can be referred to as authors, students and managers.

Recap

Persons: Represented by Profile structures. Long-term entities created when a user signs up or admin defines it.

Tracking: Encompasses Task and its sub-structure Step. Tasks are tracking devices for real-world processes; steps define actionable units within a task.

Bridging: Powered by TaskAssignment entries which link profiles to tasks. This connection gives rise to contextual roles like Student or Manager.

Roles and Context

  • Author: Emerges when a profile creates a task.
  • Student: Defined by a task assignment.
  • Manager: Assigned during task creation or via assignment metadata.

Roles are ephemeral and contextual — not inherent to the profile itself.

The simplest analogy of profiles-assignment-tasks is a new hire who needs to go through a training course. We can call that person a student during the length of the course, and cease to use the term student when the course is completed. The progress through the training course can be on a to-do list where each unit of the course is ticked off. We can model this with our system. The new hire is represented by a profile, being put on the training course is represented by an assignment and once assigned the person is referred to as a student. When the course is completed (or abandoned) the person is no longer a student. That is an easy concept to grasp, but the system can do more than this...

Abstract Flow

Profiles can also be abstract persons. This means the data structure and its data represents groups, intangible objects, & abstract concepts. This may be harder to grasp

The pupil on a training course was an easy concept to grasp. But our system can do the same for a webpage that needs to be written & published.

We use an abstract profile to represent the webpage, we have a task that has all the steps for writing and publishing the page. The task has a manager and sticking to the terminology the webpage is now the student. (It's weird to call the potential webpage a 'student', but bear with me). In this way we can use the same methods and data structure (with small changes because webpages don't have email addresses) to manage more than putting a person on a course.

Abstract Profiles

Abstract profiles represent non-human entities: webpages, organizations, ideas.

When assigned to tasks (e.g. writing and publishing a webpage), they temporarily adopt roles like Student. This allows task logic to remain consistent across humans and non-humans.

“Even abstract concepts can participate in learning, growth, and guidance — if they’re structurally represented.” -Co-pilot Microsoft

Relationships Between Profiles

The original task of the project is self management of voluntary groups. Within a large group there will be special interest groups, perhaps the group has meetings on specific subjects or in specific geographical areas. Or looking at abstract profiles there may be profiles related to webpages and others related to conferences. Imagine how to cope with various relationships between profiles and tasks. Is this profile in a group wth that profile, is this profile connected to that task, is that task connected to this other task?

Membership (“Profile A belongs to Group X”)

Affiliation (“Profile B is linked to Conference C”)

Coordination (“Profile D often works with Profile E”)

Hierarchy (“Group X is a sub-group of Network Y”)

Profiles can be related to other profiles. Tasks can be related to other tasks, and profiles can be linked to tasks even if not connected as an author, student or manager.

Analogous to assignment we have Relationship — which uses a very similar technique to allow profiles to be linked to other profiles (human or abstract), independent of tasks or to be linked to tasks independent of being the author or student or manage.It also allows tasks to be grouped or treated as subsidiaries of other tasks

This is used to represent membership, affinity, collaboration, and hierarchy.

Implementation is assumed to be in threeTables

relationships where we store the recognised ways that one profile can be related to another friend, member of, subordinate id,type,name,description

relationswhere we register who is related to whom or what is related to what. John's relation to webpage003, Consuelo's relation to John, webpage003 relation to deadline07

id,item1(FK),relationship_to(FK),item2(FK), noted_at

Example: Publishing a Webpage for a Conference

Let’s say:

  • Abstract Profile A = webpage to be published
  • Abstract Profile B = upcoming conference
  • Relationship: A showcases B
  • Abstract Profile C = publishing workflow
  • Assignment: A is assigned to C
  • Human Profile D = author, assigned to C

From this minimal setup, the system understands:

  • A is a “student” — because it is assigned to C
  • A is related to B — meaning its content is relevant to that context
  • D is responsible for publishing A

Database Schema Proposal

Table: relationships

  • id: UUID
  • relationship_type: e.g. member_of, collaborates_with
  • name: Human-readable label
  • description: Meaning and semantic role

Table: relation

  • id: UUID
  • item1: UUID of first entity
  • relationship_to: FK to relationships.id
  • item2: UUID of second entity
  • noted_at: Timestamp of relationship entry

Note: All entity IDs (profile, task, abstract item) should migrate to UUID to enable unified relational logic.
